# 04 — ملاحظات الأمان

---

## لماذا يجب حماية المثبِّت؟

المثبِّت يملك صلاحيات واسعة جداً: يكتب `.env`، يُشغِّل المايجريشن، يُنفِّذ السيدرز، ويُولِّد `APP_KEY`. أي شخص يتمكن من الوصول إليه يستطيع:

- الكتابة فوق إعدادات التطبيق.
- توجيه قاعدة البيانات نحو سيرفر خارجي.
- مسح جميع البيانات وإعادة السيدرز.

لهذا يُغلق المثبِّت نفسه تلقائياً بعد الاستخدام الأول.

---

## 1. لماذا يُستخدَم التوكن؟

التوكن هو **الحاجز الأمني الأساسي** أمام المثبِّت. بدونه، أي شخص يعرف رابط `/installer/` يستطيع الوصول إلى صفحة الإعداد.

**خصائص التوكن في هذا النظام:**

| الخاصية | القيمة |
|---|---|
| طريقة التوليد | `bin2hex(random_bytes(32))` |
| الطول | 64 حرفاً hex |
| الإنتروبيا | 256 بت (آمن تشفيرياً) |
| قيد الـ Route | `[a-f0-9]{64}` — رفض أي طلب بصيغة مختلفة |
| مدة الصلاحية | استخدام واحد فقط |
| مكان التخزين | `storage/app/installer-token.txt` |

---

## 2. لماذا يُستخدَم `hash_equals()` بدلاً من `===`؟

المقارنة العادية `===` في PHP تستغرق وقتاً متفاوتاً حسب عدد الأحرف المتطابقة في البداية. هذا يُمكِّن المهاجمين من استخدام **هجوم Timing Attack** لاستنتاج التوكن حرفاً حرفاً.

```php
// ❌ غير آمن — يمكن اختراقه بـ Timing Attack
if ($expected === $token) { ... }

// ✅ آمن — يستغرق وقتاً ثابتاً بغض النظر عن نتيجة المقارنة
if (hash_equals($expected, $token)) { ... }
```

`hash_equals()` تستغرق **وقتاً ثابتاً** دائماً — حتى لو كان الأول والأخير مختلفَين — مما يجعل هجمات Timing غير مجدية.

---

## 3. لماذا يجب أن يعمل المثبِّت مرة واحدة فقط؟

إذا بقي المثبِّت نشطاً بعد التثبيت الأول، فإنه يُشكِّل خطراً مستمراً:

- يمكن إعادة كتابة `.env` بقاعدة بيانات مختلفة.
- يمكن تشغيل المايجريشن مجدداً مما قد يُغيِّر هيكل قاعدة البيانات.
- يمكن إعادة السيدرز مما قد يُكرِّر البيانات أو يُلوِّثها.

---

## 4. لماذا `installed.lock` مهم؟

`installed.lock` هو الحارس الأساسي ضد إعادة التثبيت. يُتحقق منه في:

1. **`public/install.php`** — قبل أي معالجة من Laravel.
2. **`InstallerController::guardNotInstalled()`** — في كل طلب GET وPOST على مسارات `/installer/`.

```
أي طلب بعد وجود installed.lock → abort(404)
```

حتى لو حصل المهاجم على التوكن، لن يتمكن من تشغيل المثبِّت.

---

## 5. لماذا يجب حذف `install.php` و`installer-token.txt` بعد التثبيت؟

### `install.php`

- هو الرابط العلني المعروف للمثبِّت.
- وجوده مستمراً يدل على وجود المثبِّت في المشروع ويُشجِّع على محاولة الاستغلال.
- حتى لو كان `installed.lock` موجوداً، من الأفضل أمنياً إزالته كلياً.

### `installer-token.txt`

- يحتوي على التوكن الحقيقي.
- وجوده بعد التثبيت غير ضروري وخطير.
- إذا تمكن مهاجم من قراءة ملفات السيرفر (عبر ثغرة ما)، فإن هذا الملف يعطيه التوكن مجاناً.

> يقوم المثبِّت بحذف كليهما تلقائياً، لكن في حال فشل الحذف (بسبب صلاحيات السيرفر)، يجب الحذف اليدوي.

---

## 6. لماذا يجب أن يكون `APP_DEBUG=false` في الإنتاج؟

`APP_DEBUG=true` يُظهر في صفحات الخطأ:

- مسارات الملفات الكاملة على السيرفر.
- متغيرات البيئة (`$_ENV`).
- قيم كلمات المرور أحياناً في Stack Trace.
- هيكل قاعدة البيانات في رسائل الخطأ.

لهذا يضبط المثبِّت `APP_DEBUG=false` بشكل ثابت بغض النظر عما أدخله المستخدم.

---

## 7. لماذا يجب أن يشير الدومين إلى `/public` فقط؟

جذر مشروع Laravel يحتوي على ملفات حساسة:

```
.env                 ← بيانات الاتصال وكلمات المرور
storage/app/         ← installer-token.txt و installed.lock
vendor/              ← الحزم والمكتبات
database/            ← ملفات المايجريشن
```

إذا كان الدومين يشير إلى الجذر مباشرةً، يمكن للزوار الوصول إلى هذه الملفات.

مجلد `/public` يحتوي فقط على الملفات العامة المقصودة للمستخدمين (`index.php`, `robots.txt`, ملفات CSS/JS المُجمَّعة).

---

## 8. أخطاء أمنية شائعة يجب تجنبها

| الخطأ | الخطر | الحل |
|---|---|---|
| ترك `install.php` بعد التثبيت | إمكانية الاستكشاف | احذفه يدوياً إذا لم يُحذَف تلقائياً |
| ترك `installer-token.txt` بعد التثبيت | كشف التوكن | احذفه يدوياً |
| رفع `.env` على Git | كشف كلمات المرور | أضف `.env` لـ `.gitignore` |
| توجيه الدومين لجذر المشروع | كشف الملفات الحساسة | وجِّه إلى `/public` فقط |
| `APP_DEBUG=true` في الإنتاج | كشف معلومات النظام | المثبِّت يضبطها `false` تلقائياً |
| استخدام `shell_exec` | حقن الأوامر (Command Injection) | النظام يستخدم `Artisan::call()` فقط |
| مقارنة التوكن بـ `===` | Timing Attack | النظام يستخدم `hash_equals()` |
| الاحتفاظ بـ `installed.lock` في Git | ارتباك في البيئات المختلفة | لا يُرفع على Git |
| عدم ضبط صلاحيات `storage/` | فشل كتابة `.env` وملفات التخزين | `chmod -R 755 storage/` |
| استخدام كلمة مرور ضعيفة لقاعدة البيانات | اختراق قاعدة البيانات | استخدم كلمة مرور قوية دائماً |

---

## ملخص الضمانات الأمنية للنظام

```
✅ التوكن: 256-bit entropy — آمن تشفيرياً
✅ المقارنة: hash_equals() — مقاوم لـ Timing Attack
✅ قيد الـ Route: [a-f0-9]{64} — رفض التوكنات غير الصالحة
✅ installed.lock: يمنع إعادة التثبيت نهائياً
✅ حذف التوكن: بعد الاستخدام مباشرةً
✅ حذف install.php: بعد الاستخدام (best-effort)
✅ لا shell_exec: لا خطر حقن الأوامر
✅ APP_DEBUG=false: لا كشف للمعلومات الحساسة
✅ LOG_LEVEL=error: لا كشف للتفاصيل الداخلية في الـ log
```
